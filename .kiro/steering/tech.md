# Tech Stack & Build System

## Core Technologies

- **Language**: TypeScript (strict mode)
- **Runtime**: Bun (package manager + test runner + bundler for layer/commands)
- **Build System**: [projen](https://projen.io/) — project configuration is defined in `.projenrc.ts`
- **CDK Version**: aws-cdk-lib ^2.238.0, constructs ^10.4.5
- **jsii**: Compiles the construct library for polyglot publishing
- **Target**: ES2020, CommonJS modules

## Key Libraries

| Library | Purpose |
|---------|---------|
| aws-cdk-lib | CDK construct authoring |
| jsii | Cross-language construct compilation |
| projen | Project scaffolding and task management |
| fast-check | Property-based testing |
| eslint + prettier | Linting and formatting |
| commit-and-tag-version | Release versioning |

## Common Commands

All tasks are managed through projen. Run via `bunx projen <task>` or the npm script aliases:

```bash
# Full build (synth → compile → test → package)
bun run build

# Compile only (runs jsii)
bun run compile

# Run tests (currently runs eslint)
bun run test

# Run unit tests directly with bun
bun test

# Lint
bun run eslint

# Build the Lambda layer zip (downloads Bun binary, compiles runtime, packages)
bun run build-layer
# Or with a specific version:
bun command/buildLayer.ts 1.3.13

# Generate API docs from .jsii manifest
bun run docgen

# Synthesize projen project files after editing .projenrc.ts
bunx projen

# Install dependencies
bun install

# Upgrade dependencies
bun run upgrade
```

## Important Notes

- **Do not edit generated files directly.** Files marked with `~~ Generated by projen` must be modified via `.projenrc.ts` then regenerated with `bunx projen`.
- The `test` task currently only runs eslint. Unit tests run separately via `bun test`.
- jsii compilation is stricter than plain tsc — it enforces API compatibility rules for construct libraries.
- The layer zip is committed to `lib/` and versioned alongside the source.

## Coding Style & Conventions

### Variables & Immutability

- Always use `const`. Never use `let` unless there is no alternative (and explain why in a comment).
- Do not mutate arrays or objects. Use `.map()`, `.filter()`, spread, etc. to produce new values.
- All functions should be pure and side-effect-free where possible. When side effects are necessary (I/O, process.exit), isolate them at the edges.

### Control Flow

- Use early-return guard clauses instead of nested if/else blocks.
- Prefer extracting a helper function with guards over complex inline conditionals.
- Never use `!x` for null/undefined checks. Always use explicit `x == null` or `x != null` comparisons (loose equality intentionally catches both `null` and `undefined`).
- Never use `switch` statements or `if/else` chains for exhaustive matching. Use sequential `if` guards with an `assertUnreachable(value: never): never` call at the end. This leverages TypeScript's type narrowing — each `if` narrows the union until only `never` remains. If a case is missing, the compiler errors at the `assertUnreachable` call. Example:

```typescript
const statuses = ["ok", "error", "pending"] as const;
type Status = (typeof statuses)[number];

// Type guard — use in conditionals where you need to narrow a value
function isStatus(value: unknown): value is Status {
  return statuses.includes(value as Status);
}

// Assertion function — use when you want to throw if the value is invalid
function ensureStatus(value: unknown): asserts value is Status {
  if (statuses.includes(value as Status)) return;
  throw Error(`${value} is not valid Status. Use one of ${statuses.join()}`);
}

// Exhaustive handling with assertUnreachable
function handleStatus(props: { status: Status }): string {
  if (props.status === "ok") return "success";
  if (props.status === "error") return "failure";
  if (props.status === "pending") return "waiting";

  assertUnreachable(props.status);
}

function assertUnreachable(value: never): never {
  throw Error("Unreachable state reached!");
}
```

### Iteration

- Use `for...of` loops. Do not use `.forEach()`.

### Function Signatures

- Every function accepts a single `props` object parameter — even if there is only one argument.
- Do not destructure `props` in the function signature or body. Access fields as `props.name` directly.
- All dependencies a function needs must be passed explicitly through `props` (dependency injection via props).

### Exports & File Organization

- Only export what other modules actually need. Keep everything else private (unexported).
- Exported members go at the top of the file; private helpers go at the bottom.
- Group related types and functions together. Place types immediately before the exported function that uses them in its props or return type, followed by any private helpers that function depends on.
- Do not use unnamed default exports unless a framework explicitly requires it.
- Do not use barrel files (re-export index files).

### File Naming

- All filenames use camelCase (e.g., `buildLayer.ts`, `runtimeHelpers.ts`). Never PascalCase.

### Comments

- Do not add comments that merely restate what the code does.
- Do not add section-separator comments (e.g., `// --- Section Name ---`). Code organization should be self-evident from function grouping and file structure.
- Add comments only when there is a tradeoff, non-obvious reasoning, or a "why not the obvious approach" explanation.

### Helper Extraction & Reuse

- When you write a helper that could be reused, keep it private in the current file initially.
- Track potentially reusable helpers in `.kiro/steering/reusableHelpers.md` (create if missing). List the helper name, source file, and a one-line description.
- When the same logic is needed in a second file, extract the helper to a shared `helpers.ts` (colocated with the consumers or in a common directory) and update both call sites.

### Refactoring Discipline

- After completing each implementation step, do a quick consistency pass: verify naming, immutability, guard clauses, props pattern, and export hygiene are all followed.

### Testing

- Test only the public interface. Do not extract private functions into separate modules solely to make them testable.
- If a function is not reused outside its file, it stays private — test its behavior through the public API or integration tests instead.
- For scripts and runtimes without exports, use integration tests (e.g., spawn the process, mock external APIs, verify observable behavior).
